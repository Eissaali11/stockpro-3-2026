# خطة التكامل الشاملة
## بين قسم المنتجات والمستودعات ومخزون المندوبين والتسليم للعميل عبر تطبيق الجوال

## 1) الهدف
توحيد دورة حياة المنتج/الجهاز من لحظة إدخاله في النظام وحتى تسليمه للعميل، بحيث يتم تحديث المخزون لحظياً عبر المسح سواء من **جهاز مسح** أو **جوال المندوب**، مع ضبط صلاحيات واضح وسجل تدقيق كامل.

---

## 2) نطاق التكامل
يشمل التكامل الجهات التالية:
- قسم المنتجات
- المستودعات
- المندوبون
- المشرفون
- مدراء النظام
- العميل النهائي (مرحلة التسليم)

ويشمل العمليات التالية:
- إضافة رصيد
- إنقاص رصيد
- تحويل بين مستودع ومندوب
- سحب من مندوب
- تسليم للعميل مع التوقيع ورفع الصور ونموذج استلام PDF

---

## 3) المبادئ الأساسية للتصميم
1. **المسح هو نقطة البداية** لكل حركة مخزون.
2. **اختيار نوع العملية إلزامي** قبل المسح (إضافة/إنقاص/تحويل/سحب/تسليم).
3. **محرك حركات موحد** لجميع قنوات المسح (جهاز مسح + جوال).
4. **دفتر حركة موحد (Ledger)** هو مصدر الحقيقة، والأرصدة تُشتق منه.
5. **لا إغلاق للتسليم** دون إثبات مكتمل (توقيع + صور + PDF).

---

## 4) الأدوار والصلاحيات
### 4.1 قسم المنتجات
- إدارة بيانات المنتج المرجعية: `productId`, `barcode`, `name`, `trackingType`.
- منع تكرار أكواد المنتجات.

### 4.2 المستودعات
- تنفيذ الاستلام والصرف والاسترجاع الفعلي.
- تأكيد استلام السحب من المندوب.

### 4.3 المندوبون
- تنفيذ العمليات الميدانية عبر التطبيق (مسح، تسليم، رفع إثباتات).
- لا يملكون صلاحية اعتماد نهائي للحالات الحساسة.

### 4.4 المشرفون
- اعتماد/رفض: التسليم النهائي، السحب من المندوب، التسويات، الفروقات.
- إدارة الاستثناءات.

### 4.5 مدراء النظام
- ضبط السياسات والصلاحيات وقواعد التحقق.
- متابعة مؤشرات الامتثال والتدقيق.

---

## 5) قنوات المسح المعتمدة
### 5.1 جهاز مسح باركود
- يعمل كإدخال سريع في الشاشات التشغيلية.
- يعتمد العملية مباشرة في نمط التشغيل السريع.

### 5.2 جوال المندوب
- مسح بالكاميرا داخل التطبيق.
- يدعم التوقيع، رفع الصور، ورفع/توليد نموذج استلام PDF.

> ملاحظة تنفيذية: تطبيق الجوال سيكون **Flutter app منفصل** وسيتم تطويره في مرحلة لاحقة.

> النتيجة: كلا القناتين تغذيان نفس محرك الحركات ونفس قواعد التحقق.

---

## 6) تدفقات الأعمال (Business Flows)
### 6.1 إضافة رصيد
1. اختيار العملية: `ADD_STOCK`.
2. تحديد الجهة المستهدفة: مستودع أو مندوب.
3. مسح كود المنتج.
4. إنشاء حركة Ledger موجبة وتحديث الرصيد.

### 6.2 إنقاص رصيد
1. اختيار العملية: `DEDUCT_STOCK`.
2. تحديد الجهة المستهدفة.
3. مسح كود المنتج.
4. التحقق من توفر الرصيد.
5. إنشاء حركة Ledger سالبة وتحديث الرصيد.

### 6.3 تحويل من المستودع إلى المندوب
1. اختيار العملية: `TRANSFER_TO_TECHNICIAN`.
2. مسح المنتج.
3. إنشاء حركتين مترابطتين بنفس `operationId`:
   - خصم من المستودع.
   - إضافة لعهدة المندوب.

### 6.4 سحب من المندوب
1. إنشاء طلب سحب من المشرف.
2. المندوب يسلم عبر المسح.
3. المستودع يؤكد الاستلام.
4. اعتماد نهائي من المشرف.
5. إغلاق الطلب أو فتح محضر فرق عند النقص.

### 6.5 تسليم للعميل عبر جوال المندوب
1. المندوب يفتح أمر تسليم في التطبيق.
2. مسح المنتج المراد تسليمه.
3. تسجيل بيانات الاستلام.
4. توقيع العميل إلكترونياً.
5. رفع صور الإثبات.
6. رفع ملف PDF (أو توليده من النموذج).
7. إرسال للمشرف للاعتماد النهائي.
8. عند الاعتماد: تحديث الحالة إلى `DELIVERED` وإغلاق العملية.

## 7) نموذج البيانات المقترح
## 7.1 الجداول الأساسية
- `products` (مرجع المنتجات)
- `stock_balances` (الرصيد الحالي حسب الجهة)
- `stock_ledger` (كل الحركات)
- `scan_events` (تفاصيل كل مسحة)
- `delivery_receipts` (إثباتات التسليم)
- `recall_orders` و `recall_order_items` (سحب من المندوب)

### 7.2 حقول محورية مقترحة
#### في `stock_ledger`
- `id`
- `operationId`
- `operationType`
- `productId`

- `deltaQty` (موجب/سالب)

#### في `scan_events`

## 8) حالات العملية (Lifecycle)
- `NEW`
- `IN_WAREHOUSE`
- `WITH_TECHNICIAN`
- `PENDING_DELIVERY`
- `PROOF_SUBMITTED`
- `PROOF_APPROVED` / `PROOF_REJECTED`
- `DELIVERED`
- `RECALLED` (عند السحب)

---

## 9) قواعد التحقق والحوكمة
1. منع الرصيد السالب.
2. منع تكرار المسح عبر `idempotencyKey`.
3. للأصناف المتسلسلة (Serial): لا يمكن أن يكون نفس السيريال في عهدة جهتين.
4. أي تسوية أو فرق كبير يتطلب اعتماد مشرف.
5. لا يمكن إغلاق التسليم دون:
   - توقيع عميل.
   - صورة إثبات واحدة على الأقل.
   - ملف استلام PDF.
6. كل العمليات تحفظ في سجل تدقيق غير قابل للتعديل.

---

## 10) واجهات تشغيل مقترحة (MVP)
1. شاشة موحدة للحركة: (إضافة/إنقاص + اختيار الجهة + مسح + تأكيد).
2. شاشة تحويل مستودع ⇄ مندوب.
3. شاشة سحب من مندوب (طلب، تنفيذ، استلام، اعتماد).
4. شاشة تسليم العميل في تطبيق المندوب (توقيع + صور + PDF + إرسال اعتماد).
5. لوحة إشراف للاعتمادات والتنبيهات والفروقات.

---

## 11) مراحل التنفيذ
### المرحلة 1: نموذج البيانات والمحرك
- تجهيز الجداول والعلاقات.
- بناء محرك حركة موحد.

### المرحلة 2: العمليات الأساسية بالمسح
- إضافة/إنقاص/تحويل.
- تحديث الرصيد الفوري.

### المرحلة 3: سحب المخزون من المندوب
- طلب سحب + تنفيذ + اعتماد.

### المرحلة 4: التسليم للعميل عبر الجوال
- توقيع + صور + PDF + اعتماد مشرف.

### المرحلة 5: التقارير والحوكمة
- لوحات متابعة.
- مؤشرات الأداء والتنبيهات.

---

## 12) مؤشرات النجاح (KPIs)
- دقة المخزون بين النظام والواقع.
- نسبة العمليات المكتملة بإثبات صحيح.
- زمن اعتماد المشرف للحركات الحساسة.
- عدد الفروقات الشهرية ومعدل إغلاقها.

---

## 13) ملاحظات تنفيذية
- يوصى باعتماد **Ledger-First Architecture**: أي تحديث رصيد يجب أن يمر عبر سجل الحركة أولاً.
- يوصى بإضافة دعم **Offline Queue** في تطبيق الجوال للمناطق ذات الإنترنت الضعيف.
- يوصى بتوحيد `reasonCode` عبر جميع الوحدات لتحسين التقارير.

---

## 14) التنفيذ الحالي (المرحلة الأولى)
تم تنفيذ نقطة API موحدة للمسح في الخادم:

- `POST /api/inventory-scan/execute`
- المصادقة: `requireAuth`
- API جاهز لمصدرين: `scanner` أو `mobile`
- التنفيذ الحالي في الواجهة: `scanner` (ويب/قارئ باركود)
- مصدر `mobile` محجوز لتطبيق Flutter المنفصل (لاحقاً)

### 14.1 العمليات المدعومة في نقطة المسح
- `ADD_STOCK`
- `DEDUCT_STOCK`
- `TRANSFER_TO_TECHNICIAN`
- `WITHDRAW_FROM_TECHNICIAN`

### 14.2 أمثلة Payload

#### إضافة رصيد لمستودع
```json
{
   "source": "scanner",
   "operationType": "ADD_STOCK",
   "itemCode": "rollPaper",
   "packagingType": "box",
   "quantity": 5,
   "ownerType": "warehouse",
   "ownerId": "<warehouse-id>",
   "reasonCode": "initial_stock"
}
```

#### إنقاص رصيد من مندوب
```json
{
   "source": "scanner",
   "operationType": "DEDUCT_STOCK",
   "itemCode": "rollPaper",
   "packagingType": "unit",
   "quantity": 2,
   "ownerType": "technician",
   "ownerId": "<technician-id>",
   "reasonCode": "customer_usage"
}
```

#### تحويل من مستودع إلى مندوب
```json
{
   "source": "scanner",
   "operationType": "TRANSFER_TO_TECHNICIAN",
   "itemCode": "rollPaper",
   "packagingType": "box",
   "quantity": 1,
   "warehouseId": "<warehouse-id>",
   "technicianId": "<technician-id>",
   "reasonCode": "warehouse_dispatch"
}
```

### 14.3 ملاحظات تنفيذية مهمة
- البحث عن المنتج في التنفيذ الحالي يتم عبر `itemType.id` أو الاسم (`nameAr` / `nameEn`).
- عند توفير `idempotencyKey` يتم منع إعادة تنفيذ نفس عملية المسح.
- يتم تحديث مخزون **entries الجديد** مع مزامنة الجداول القديمة للحفاظ على التوافق الخلفي.
- يتم تسجيل كل عملية في:
   - `stock_movements` كسجل حركة.
   - `system_logs` كسجل تدقيق مفصل.
